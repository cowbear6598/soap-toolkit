# 風格 Review 規則

## 檢查範圍

僅檢查本次有異動的程式碼。

## 檢查項目

### 命名

- 函數和變數命名具描述性，能從名稱理解用途
- 避免無意義的縮寫（`usr`、`tmp`、`val` 等，除非是業界慣例）
- 避免 magic value（直接寫死的數字或字串），應該用常數或 enum 賦予語意
- 布林值命名應能讀成一句話（`isActive`、`hasPermission`、`canEdit`）

### 一致性

- 同一個專案內命名風格保持一致（不混用 camelCase 和 snake_case）
- 同類型的東西用同樣的命名模式（如果其他 handler 都叫 `handleXxx`，新的也該如此）
- 與既有程式碼風格保持一致，不在局部引入新風格

### 可讀性

- 避免過長的鏈式呼叫，適時用有意義的中間變數增加可讀性
- 條件判斷過於複雜時，抽成有語意的變數或方法（`const isEligible = ...` 比一長串 `&&` `||` 好讀）

## 輸出格式

```markdown
## 風格 Review 結果

### ✅ 符合規範
- [列出符合的項目]

### ⚠️ 建議改善
| 檔案 | 行數 | 問題 | 建議 |
|------|------|------|------|

### ❌ 違反規範
| 檔案 | 行數 | 問題 | 建議 |
|------|------|------|------|
```

## 注意事項

- 只報告實際問題，不要過度挑剔
- 邊界情況給建議而非強制要求
