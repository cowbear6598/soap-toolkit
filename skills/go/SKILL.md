# Go 程式碼風格指南

在所有 Go 程式碼操作時套用此專案的程式碼風格規範。

## 核心原則

1. 簡短、易讀、有點彈
2. 不需要過多的防衛性編程
3. 專注於簡潔和清晰，而非複雜性
4. 查看 [examples.md](examples.md) 了解優良示範

## 指導方針

### 實體定義

- 將 `foreignKey` 欄位放在實體結構定義的**最尾端**
- 保持實體結構乾淨且有組織

### 資料驗證

- **使用 GORM binding 標籤**進行驗證 - 不要另外撰寫 validator 程式碼
- 善用 GORM 內建的驗證功能，而不是自訂驗證邏輯

### API 設計

- **請求表單**：所有表單欄位名稱使用 PascalCase，與實體欄位名稱一致
- **API 變動**：當 API 有變動時，務必同步更新 `docs/` 中的文件

### 註解

僅對**複雜邏輯**加上註解：
- 計算複雜度分數：每個 `if`、`for` 或 `switch case` = 1 分
- 巢狀結構 = 分數加倍
- **只在分數超過 6 分時才加上註解**
- 註解應該解釋**為什麼**（原因），而不是**做什麼**（動作）

### 測試

- **測試命名**：使用 `Test_Success_When_{$情境}` 和 `Test_Fail_When_{$情境}` 格式
- **檔案分離**：
  - 核心邏輯測試：`*_test.go`
  - 驗證器測試：`*_validator_test.go`
- **重點**：針對條件邏輯和真實使用情境撰寫測試
- **避免**：過度的邊界測試
- **要求**：每個功能都必須有對應的測試
- **不要使用** `go build` 指令 - 使用測試來驗證程式碼正確性

## 專案結構參考

- `entity/` - 資料庫實體
- `usecase/` - 核心業務邏輯
- `controller/` - API 端點
- `error/` - 錯誤定義
- `repository/` - 資料庫操作
- `request/` - API 請求 DTO
- `response/` - API 回應 DTO
- `pkg/` - 第三方服務
- `utils/` - 共用功能

## 技術堆疊

- **ORM**：GORM
- **資料庫**：PostgreSQL
- **框架**：Gin
- **架構**：RESTful API
