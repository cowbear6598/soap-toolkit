## Step 1：取得實作輸入來源

實作不再限定只能從 `frontend.md` / `backend.md` 驅動，輸入來源依下列優先順序嘗試，第一個成立就採用：

### 優先級規則（來源 1 vs 來源 2）

- **預設 plan 檔 > context**：因為 plan 檔是刻意產出的設計結論，通常經過比對話更完整的思考。
- **例外：context 明確要求忽略 plan**：若 context 中使用者明確指示「忽略 plan」「以對話為主」「跳過計畫書」「不要看 tasks」等語意，則改以 context 為主並忽略 plan 檔。
- **歧義：context 疑似修改 plan**：若偵測到 context 有修改 plan 檔內容的跡象（例如使用者剛剛說「把 Phase 2 的 A 改成 ...」）但沒明確要求忽略 plan，**先反問使用者**要以哪個為準，不要自行判斷。

### 來源 1：Plan 檔

檢查專案內是否存在：

- `tasks/frontend.md`（前端計畫書）
- `tasks/backend.md`（後端計畫書）

讀取 plan 檔後**先驗證格式**：至少要能解析出一個 Phase 與其下的大項目。若檔案存在但空、格式錯誤、或無任何 Phase，視為**該 plan 檔無效**，繼續嘗試下一個 plan 檔；若兩份 plan 檔都無效，轉往來源 2（Context）。明確告知使用者哪個 plan 檔被判定為無效及原因（例如「`tasks/frontend.md` 內容為空」或「`tasks/backend.md` 解析不到任何 Phase 標題」）。

只要任一個有效的 plan 檔存在，就採用「Plan 檔」來源，照既有的 Phase 結構走：

- `frontend.md` 和 `backend.md` 都有效就兩條線都處理；只有其中一個有效就只處理那一條。
- 同時讀取 `tasks/userflow.md` 作為背景知識傳給每個 Agent。若 `userflow.md` 不存在就略過，不視為錯誤。

### 來源 2：Conversation context

當 `tasks/frontend.md` 與 `tasks/backend.md` 都不存在（或都無效）時，檢查目前對話 context 是否含有「可實作資訊」。

#### 「可實作資訊」判準

context 必須**能同時回答以下三個問題**才算具備「可實作資訊」：

1. **要改哪些檔案或新增哪些檔案**（具體檔案路徑或檔名）
2. **改什麼**（具體行為、邏輯、規格）
3. **為什麼改 / 驗收條件**（至少有其一）

判定規則：

- 若三項都能從 context 回答 → 進入實作。
- 若任一項缺 → 轉往來源 3 反問。

#### 具體範例

- **範例 A（充分）**
  - 對話片段：使用者剛跑完 `/soap-dev:review`，review 結果列出了 3 個「必須修正」項目，每項都有檔案路徑與問題描述。使用者接著說：「把『必須修正』全修了」。
  - 判定結果：**進入實作**。
  - 理由：(1) 檔案可從 review 結果抽出；(2) 改什麼由 review 項目描述；(3) 為什麼改＝review 已指出的問題。三問皆可回答。

- **範例 B（不充分）**
  - 對話片段：助理貼了一段 review 或建議，使用者只回「OK」或「看起來不錯」。
  - 判定結果：**反問**。
  - 理由：使用者沒表達要修或不修，也沒指定範圍；意向不明，三問中的「改什麼」無法確定是否要改全部。

- **範例 C（充分）**
  - 對話片段：前一輪助理輸出 bug fix 建議，明列「修改 `src/foo.ts` 的 `parseX()` 函式，讓它處理空字串回傳 null」並附驗收條件，使用者回「好」。
  - 判定結果：**進入實作**。
  - 理由：(1) 檔案 `src/foo.ts` 明確；(2) 改什麼＝`parseX()` 空字串處理；(3) 驗收條件助理已列。三問皆可回答，使用者「好」視為同意。

- **範例 D（不充分）**
  - 對話片段：使用者只說「這個專案有幾個錯誤我想修一下」，沒列檔案、沒描述哪個錯誤。
  - 判定結果：**反問**。
  - 理由：沒有具體檔案、也沒說改什麼；三問全不可答。

符合「可實作資訊」判準時，將這些項目整理成一份臨時實作清單，每一項至少包含：

- 要改的檔案
- 要做什麼改動
- 對應的使用者意向 / 驗收條件（例如：「使用者同意修 review 必須修正項」「使用者指定優先處理空字串問題」）

同樣要讀取 `tasks/userflow.md` 作為背景知識傳給每個 Agent，若不存在則略過。

**明確說明**：不再要求只有 `frontend.md` / `backend.md` 才能 implement。只要 context 通過上述三問判準，就直接進入實作流程，不反問使用者、也不要求使用者先去產 plan。

### 來源 3：反問使用者

當 Plan 檔不存在（或皆無效）、且 context 也沒有可實作資訊時，不得自行推測要做什麼。直接以一句話詢問使用者想實作什麼，等待使用者補充。

#### 收到使用者回覆後的銜接

- 把使用者回覆內容視為「新增的 conversation context」，**重新跑一次 Step 1 判定**（從來源 1 開始：先重新檢查 plan 檔是否變為有效，再用來源 2 的三問判準檢查新 context）。
- 若回答仍不足以通過「可實作資訊」判準（三問中任一缺），**再次反問**。
- **最多反問 2 次**。仍不足則停止並告訴使用者：「目前輸入不足以開始實作，請使用 `/soap-dev:plan` 產出計畫書後再試」。

## Step 2：按 Phase 依序執行

### 來源為 Plan 檔時

計畫書的結構是用 Phase 分組，每個 Phase 內有大項目（A. B. C. ...），大項目下是具體的 Todo。

執行規則：

- **前端和後端是兩條獨立的執行線**：各自按自己的 Phase 順序推進，互不等待。前端 Phase 1 完成就直接進前端 Phase 2，不需要等後端。
  - 澄清：「互不等待」是指不同「Phase 進度」上互不阻塞；但若某個大項目在 prompt 中明確標註「需等待前端 / 後端 Phase N 完成」，則遵守該明確標註。
- **同一條線內，Phase 必須依序完成**：Phase 1 的所有大項目全部完成後，才能推進到 Phase 2。
- **同一個 Phase 內的並行規則**：
  - 如果 Phase 標題標註了「可並行」，該 Phase 內的大項目可以同時啟動多個 Agent。
  - 如果 Phase 標題沒有標註「可並行」，該 Phase 內的大項目必須依序執行（A 完成後才啟動 B，B 完成後才啟動 C）。

### 來源為 Conversation context 時

Context 來源沒有 Phase 結構，處理方式如下：

- 把 context 中所有要修的項目視為「單一 Phase」。
- 判斷這個單一 Phase 內是否可並行：
  - 檢查各項目是否有**明顯的共同檔案修改衝突**。
  - **「明顯衝突」判定標準**：兩個大項目的 Todo 中是否包含**同一檔案的修改**。若有任一檔案被兩個以上大項目同時宣告要改 → 視為衝突，改依序執行。**檔案層級衝突已經足夠，不需要再細分到函數/區塊**。
  - 沒有衝突 → 視為「可並行」，按並行規則同時啟動多個 Agent。
  - 有衝突 → 改為依序執行，一個做完再做下一個。
- 其他 Phase 規則（前後端兩線、同 Phase 並行規則）不套用到 context 來源；那些規則只對 Plan 來源有效，照舊保留。

## Step 3：啟動 Agent

每個大項目（A. B. C. ...）啟動一個獨立的 Agent。使用通用 agent（不指定 subagent_type），model 設為 sonnet。

### 嚴格遵守的原則

- **一個 Agent 只負責一個大項目**。絕對不要把多個大項目塞進同一個 Agent。
- **不要假設 Agent 知道任何背景**。Agent 是全新的，沒有讀過計畫書、沒有看過 codebase，所有需要的資訊都必須寫進 prompt。

### 傳給 Agent 的 prompt 必須包含以下三個部分

1. **該大項目的完整 Todo 內容**：把計畫書中該大項目底下的所有 Todo 原封不動貼進去，讓 Agent 知道要做什麼。
2. **userflow.md 的相關情境**：從 userflow.md 中擷取與該大項目相關的使用者情境，讓 Agent 理解為什麼要做這些事、使用者會怎麼操作。
3. **實作規則**：把下方「實作規則」區塊的內容完整貼進 prompt。

### 來源為 Conversation context 時的補充

- 「大項目」的定義改為「context 清單中的每一項（或每一個可獨立完成的子任務）」。一個 Agent 仍然只負責一個這樣的大項目。
- 傳給 Agent 的 prompt 第一部分（Todo 內容）改成「從 context 整理出的該項目 todo 敘述」，需具體到檔案與要做的改動，不可只丟模糊描述。
- 第二部分（userflow.md 相關情境）若 `userflow.md` 不存在就省略，不視為錯誤；若存在則同樣擷取相關段落。
- 第三部分（實作規則）照舊完整貼入。

## Step 4：Phase 完成後驗證

每個 Phase 的所有 Agent 都完成後（不論是並行還是依序），執行測試驗證。對 context 來源整理出的「單一 Phase」同樣要跑這一步。

1. **偵測專案類型並執行對應測試指令**：
   - 找到 `*.csproj` → 執行 `dotnet test`
   - 找到 `package.json`：
     - 先查看 `scripts.test`。若存在且不是預設的 `"exit 1"`，照 `npm test` / `bun test` 執行。
     - 若 `package.json` 存在但 `scripts.test` 不存在或是預設的 `exit 1`，視為「無測試設定」，按下方「完全找不到任何測試設定」規則處理。
   - 找到 `go.mod` → 執行 `go test ./...`
   - 以上都沒有 → 按下列清單搜尋測試設定檔：
     - `jest.config.*`
     - `vitest.config.*`
     - `pytest.ini`
     - `pyproject.toml` 的 `[tool.pytest.ini_options]`
     - `karma.conf.*`
     - `Makefile` 的 test target
     - `.github/workflows/*.yml` 中的 test step
   - 找到任一測試設定 → 照該工具執行對應測試指令。
   - **完全找不到任何測試設定**：直接跳過測試驗證，在回報中明確說「此專案未偵測到測試框架，跳過 Phase 驗證；使用者需自行手動測試」。不要無限迴圈、也不要自己產生測試檔。
2. **測試通過才進入下一個 Phase**。
3. **測試失敗就修正，修完再跑測試，重複直到全部通過**。只有測試全部通過後，才能推進到下一個 Phase。
4. **來源為 Context 時的收斂規則**：當輸入來源為 Context 時，該「單一 Phase」完成測試驗證後即結束整個流程，不尋找下一個 Phase。若測試失敗則循環修正，直到通過或使用者決定放棄。

## 實作規則

以下規則必須完整傳給每一個 Agent：

### 實作前

- 善用搜尋工具了解程式碼的依賴關係和既有架構。
- 計畫書提供方向，但實作時會遇到計畫書沒提到的細節，要自己查閱 codebase 確認再動手。
- 如果發現計畫書有不合理的地方，以實際 codebase 為準，不要盲目照抄計畫書。

### 實作中

- 不要產出任何 summary、plan、task 等 markdown 檔案。
- 專注在寫程式碼，不是寫文件。
