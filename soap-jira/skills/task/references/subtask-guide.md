# 子任務拆解指南

## 核心原則

### Feature 層級思考

子任務必須以**功能面**為單位，而非技術實作細節。

- 正確：「完成使用者註冊功能」— 包含該功能所需的前後端實作
- 錯誤：「建立 UserController」「撰寫 RegisterService」— 這是實作細節，不是任務

每個子任務應該代表一個可交付的功能切片，讓開發者能從頭到尾完成一個完整的功能點。

### 禁止項目

- **嚴禁**建立測試任務 — 測試（單元測試、整合測試）是每個功能子任務的內建職責，不獨立成票
- **嚴禁**建立驗收任務 — 驗收由 QA 流程處理，不在開發子任務範疇內
- **不需要** UI 切版任務 — UI 已由切版團隊製作完成，開發只需 Merge Branch 並整合

## 拆解維度

根據 Story 的實際需求，從以下維度靈活組合，不要硬套所有維度：

| 維度 | 適用情境 | 範例 |
|------|----------|------|
| 資料層 | 需要新建或調整資料表、欄位、Migration | 建立 XX 資料表與 Migration |
| 後端功能 | API、商業邏輯、服務整合 | 實作 XX 功能的 API |
| 前端功能 | 頁面邏輯、狀態管理、API 串接 | 整合 XX 頁面與 API 串接 |
| 排程/背景作業 | 定時任務、Queue、Event Handler | 實作 XX 排程處理 |
| 第三方整合 | 外部 API、SDK、Webhook | 串接 XX 第三方服務 |
| 跨服務溝通 | SignalR、Event Bus、Message Queue | 實作 XX 即時通知功能 |

**關鍵**：不是每個 Story 都需要涵蓋所有維度。如果一個 Story 只涉及後端邏輯調整，就只拆後端相關的子任務。讓需求決定維度，而不是維度決定需求。

## 拆解步驟

1. **分析需求**：從 Story 的 summary 和 description 中識別所有功能點
2. **比對現有實作**：根據 codebase 搜尋結果，了解哪些部分已存在、哪些需要新建或修改
3. **識別依賴**：找出功能點之間的依賴關係，決定是否需要拆分或合併
4. **組成子任務**：將功能點按 Feature 層級組成子任務，確保每個子任務有明確的範圍和目標
5. **檢查完整性**：確認所有子任務加總能完整覆蓋 Story 的需求

## 特殊子任務

以下子任務**只在 Story 的 description 中明確提到時**才建立，不要自行推斷或假設需要：

- **HttpLog**：Story 明確要求記錄 HTTP 請求/回應
- **文件更新**：Story 明確要求更新 API 文件、規格書等專案文件

如果 Story 沒有提到，就不建。這兩個子任務有固定的格式和驗證標準，與業務功能邏輯無關，因此獨立拆出。

## 範例

**Story**：身為使用者，我希望能透過註冊功能來使用該產品的所有功能

**分析後的子任務**：
1. 建立 User 相關資料表 — 資料層，需先完成才能進行後續開發
2. 實作註冊功能與前端整合 — 後端 API + 前端頁面串接，一個完整的功能切片
