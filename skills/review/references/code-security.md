# 安全性 Review 規則

## 檢查範圍

僅檢查本次有異動的程式碼。

### 注入防護

- 資料庫查詢是否使用參數化（禁止字串拼接 SQL）
- 是否有 Command Injection 風險（將用戶輸入傳入系統指令）
- 動態生成的 HTML 是否有 XSS 風險

### 認證與授權

- API 端點是否有適當的認證保護
- 權限檢查是否完整（不只檢查登入，還要檢查角色/權限）
- 敏感操作是否有額外驗證

### 敏感資料

- 不在前端儲存敏感資訊（token 除外）
- 硬編碼的 API key、密碼、token、secret
- 日誌是否記錄敏感資訊（密碼、token、個資）
- API 回應是否過度暴露資料（回傳了不必要的欄位）
- console.log / print 是否洩漏敏感資料

### 邏輯漏洞

- 競爭條件（Race Condition）風險
- 業務邏輯繞過可能性
- 數值溢位/下溢風險
- 空值處理是否完整

### 錯誤處理

- 例外是否洩漏堆疊資訊或內部細節給用戶
- 錯誤訊息是否過於詳細，讓攻擊者能推測系統架構

### 資料流追蹤

對於用戶輸入，追蹤其流向：每一個接收用戶輸入的入口，追蹤到它最終被使用的地方，確認中間是否有驗證和消毒。

## 嚴重程度定義

| 等級 | 定義 | 範例 |
|------|------|------|
| Critical | 可直接利用造成重大損害 | SQL Injection、認證繞過 |
| High | 需要特定條件但影響大 | 權限提升、敏感資料洩漏 |
| Medium | 有限度的安全影響 | 資訊洩漏、弱加密 |
| Low | 最佳實踐建議 | 過度錯誤訊息、缺少 rate limit |

## 注意事項

- 優先報告高風險問題
- 提供具體的修復建議和程式碼範例
- 以攻擊者視角思考
- 金流、認證相關程式碼要特別仔細
