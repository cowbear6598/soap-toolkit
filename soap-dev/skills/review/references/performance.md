# 效能 Review 規則

## 檢查範圍

僅檢查本次有異動的程式碼。

## 檢查項目

### 資料庫效能

N+1 query（在迴圈中逐一查詢資料庫）、缺少必要索引、不必要的 JOIN、未分頁的大量查詢、SELECT * 而非選擇實際需要的欄位，都會造成資料庫負擔過重，在資料量增長後迅速成為瓶頸。

### 演算法與資料結構

不必要的巢狀迴圈（O(n²) 問題可以用 O(n) 解決）、低效排序、重複計算（同樣的結果應該快取而非重複運算）、應該用 HashSet/Dictionary 查找卻用 List 線性掃描，這些問題在資料量大時會造成顯著的效能下降。

### 記憶體與資源

記憶體洩漏（事件未取消訂閱、大物件未適當釋放）、不當持有大物件、快取策略問題（無上限快取、永不過期的快取）、未使用 IDisposable/using 管理非受控資源，都會讓記憶體持續增長，最終導致服務不穩定。

### I/O 與網路

同步阻塞呼叫（應改用 async/await）、未批次處理的 API 請求（在迴圈中逐一呼叫外部服務）、不必要的序列化/反序列化、缺少超時設定（可能導致請求無限等待），這些問題會造成執行緒阻塞或不必要的延遲。

## 回報方式

發現問題時，說明：
- 屬於哪一類效能問題（資料庫、演算法、記憶體、I/O）
- 為什麼這段程式碼有效能風險
- 在什麼情況下問題會顯現（例如資料量大、高並發、長時間運行）
- 提出具體的改善方案
