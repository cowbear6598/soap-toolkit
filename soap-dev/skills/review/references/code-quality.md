# 品質 Review 規則

## 檢查範圍

僅檢查本次有異動的程式碼。

## 檢查項目

### 單一職責

一個 class 或方法是否只做一件事。如果你很難用一句話描述它的職責，就代表它做太多了。

### 錯誤處理

- 不要用無意義的 try-catch 過度保護，try-catch 應該只出現在有實際失敗可能的邊界（第三方服務、I/O、網路請求）
- 其餘情況讓錯誤自然浮現，提前暴露問題比吞掉錯誤好
- 錯誤處理應該有明確的意圖，不是「以防萬一」

### 抽象合理性

- 不要為了一次性的操作建立抽象（不需要為只用一次的邏輯寫 helper）
- 反過來，如果同樣的邏輯出現多次，就該抽取
- 抽象的命名和層級應該讓使用者不需要看實作就能理解用途

### 資料流清晰度

- 資料的流向應該是可追蹤的，不要讓資料在多個地方被隱式修改
- 偏好不可變的資料結構，減少副作用
- 避免用全域狀態或隱式的依賴傳遞資料

### 非同步

- 偏好 async/await，避免阻塞式的等待
- 不要混用同步和非同步模式

### Fallback 必要性

列出本次異動程式碼中**所有** fallback / defensive code，逐一標註必要性。檢查目標：

- try-catch 包覆
- null / undefined / 空值的 fallback（例如 `value ?? default`、`if (!x) return [...]`）
- 預設值兜底（找不到資料給預設值）
- 重試邏輯 retry / backoff
- TypeScript 型別系統已保證的防衛

每一項標註以下其中之一：

- **必要** — 真實 I/O 邊界（第三方服務、網路、檔案系統）、業務需要的 UX 救援
- **可移除** — 型別系統已保證、不會發生的情況、過度防衛
- **需確認** — 不確定，建議跟使用者討論

**規則精神**：標註必要性時，必要性判定必須**從正規使用路徑反推**。

- ✅ 允許的論證：「正規 UI 流程下會送出 X → 必要」「真實 I/O 邊界會失敗 → 必要」
- ❌ 禁止的論證：用假設性 client / 假設性威脅 / 想像中的外部腳本當理由（除非系統實際存在這些 client）。「CLI 可能...」「測試工具可能...」「萬一有人...」這類論證一律標為「可移除」或「需確認」，不能標「必要」。

判定流程：
1. 從正規使用路徑可達？→ 否 → 標「可移除」（死代碼）
2. 可達 → 看是否有業務必要性 → 必要 / 可移除 / 需確認

此檢查結果在 review 輸出時**獨立成一個區塊**，不混進「建議改善」或「違反規範」表格。

## 輸出格式

```markdown
## 品質 Review 結果

若沒有任何問題（包含 Fallback 必要性區塊也無內容），回報單行「無問題」即可，不列檔案清單或表格。

### ⚠️ 建議改善
| 檔案 | 行數 | 問題類型 | 問題描述 | 建議 |
|------|------|----------|----------|------|

### ❌ 違反規範
| 檔案 | 行數 | 問題類型 | 問題描述 | 建議 |
|------|------|----------|----------|------|

### 🛡 Fallback 必要性盤點
| 檔案 | 行數 | Fallback 類型 | 必要性 | 說明 |
|------|------|---------------|--------|------|
```
