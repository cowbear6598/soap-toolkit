從使用者操作路徑出發，追蹤呼叫鏈，定位 bug 的根因。

**重要：bug 模式只調查，不修復。** 你可以讀檔案、搜尋程式碼、啟動 agent 調查，但絕對不能修改任何程式碼。調查完成後產出根因分析報告，修不修、怎麼修由使用者決定。

**為什麼不從錯誤訊息直接 grep 程式碼？** 在複雜專案中（例如一個 repo 裡有多個前端和後端子專案），直接 grep error message 可能會找到錯的專案或錯的路徑。正確的做法是先理解使用者的操作流程，確認是哪個前端觸發、打到哪個後端，再沿著這條路徑追蹤。

**不需要主動問使用者額外問題。** 使用者提供了什麼就用什麼。如果他沒說操作步驟，就從 codebase 去追蹤還原。

## 流程

### Step 1：探索專案結構

使用 Agent tool（`subagent_type: Explore`，`model: sonnet`）了解整個專案的架構：
- 有哪些子專案（前端、後端）
- 各自用什麼框架 / 技術棧
- 子專案之間的關係（哪個前端打哪個後端的 API）

這一步是為了在後續追蹤時能走對路徑，不會追到錯的子專案裡。

### Step 2：還原使用者操作路徑

根據使用者提供的 bug 描述，從使用者視角還原操作路徑：
- 這個功能在哪個前端專案
- 使用者的操作流程是什麼（點了什麼按鈕 → 觸發什麼動作 → 呼叫什麼 API）
- 如果使用者沒提供操作步驟，從功能名稱或錯誤位置反推這個功能的使用流程

使用 Agent tool（`subagent_type: Explore`，`model: sonnet`）搜尋 codebase 來追蹤這條路徑。

### Step 3：沿路徑追蹤呼叫鏈

順著 Step 2 還原的操作路徑，追蹤完整的呼叫鏈。可以同時啟動多個 Agent 平行調查不同段落：

- **前端段**：UI 元件 → 事件處理 → API 呼叫
- **API 段**：路由 → Controller → Service
- **資料段**：資料庫查詢 → 資料轉換 → 回傳

每個 Agent 使用 `subagent_type: Explore`，`model: sonnet`，prompt 要明確告訴 agent：
1. 你在追蹤哪條路徑的哪一段
2. 你要找什麼（相關的檔案、函數、API 端點）
3. 特別注意哪些可能出錯的地方（例如：型別轉換、null check、邊界條件、非同步處理）

### Step 4：根因分析

彙整所有 Agent 回報的結果，定位根因。如果初步調查不夠明確，可以再啟動 Agent 深入調查特定可疑區域。

### 輸出格式

```markdown
# Bug 調查報告

## 問題描述
[使用者回報的問題，一句話]

## 使用者操作路徑
1. 使用者在 [哪個頁面/功能]
2. 執行 [什麼操作]
3. 預期 [應該發生什麼]
4. 實際 [發生了什麼]

## 呼叫鏈追蹤
[前端入口] → [API 呼叫] → [後端處理] → [資料層]

標示每一段的關鍵檔案和行數，用 file_path:line_number 格式。

## 根因
[明確指出問題出在哪裡、為什麼會出錯]

## 影響範圍
- 直接影響：[這個 bug 直接導致的問題]
- 潛在影響：[同樣的根因可能影響的其他功能]

## 建議修復方向
[不做修復，但給出修復的方向性建議]
```
