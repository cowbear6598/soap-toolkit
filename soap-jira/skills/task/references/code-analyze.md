# 程式碼分析流程

## 分析目標

深入理解 codebase 搜尋階段找到的程式碼，建立對現有實作的完整認知，為子任務拆解提供具體的技術依據。

## 分析流程

### Step 1：閱讀核心程式碼

針對搜尋階段找到的**直接相關檔案**，使用 Read 工具逐一閱讀：

- 與 Story 實體直接相關的 Model / Entity
- 現有的 Controller、Service、Repository
- 前端的 Component、Composable、API 模組

重點關注：
- 資料結構和欄位定義
- 現有的 API 端點和商業邏輯
- 前端頁面的元件組成和資料流

### Step 2：追蹤依賴鏈

從核心程式碼出發，追蹤其 `using`、`import` 語句，閱讀直接依賴的檔案：

- 被注入的 Service 和 Repository
- 繼承的基底類別或實作的介面
- 引用的共用 Composable 和 Utility
- MediatR 的 Request / Handler 對應關係

**追蹤邊界**：
- 只追蹤與 Story 功能相關的依賴，忽略通用基礎設施（如 Logger、BaseController）
- 最多追蹤兩層依賴（核心 -> 直接依賴 -> 間接依賴），避免無限展開
- 若遇到第三方套件邊界，記錄套件名稱即可，不需深入

### Step 3：分析相似功能

閱讀搜尋階段找到的**相似功能範例**，重點理解：

- 實作模式：該功能從 Controller 到 Service 到 Repository 的完整流程
- 檔案結構：每個功能涉及哪些檔案，如何組織
- 命名慣例：類別、方法、檔案的命名規律
- 前後端串接方式：API 呼叫、資料格式、錯誤處理

這些資訊用於：
- 預估新 Story 需要建立哪些檔案
- 作為子任務 description 中的參考檔案
- 確保新實作與現有架構一致

### Step 4：彙整分析結論

將分析結果整理為以下結構，直接支撐子任務拆解：

1. **資料層現況**：現有的資料表、需要新增或修改的欄位
2. **後端現況**：現有的 API、Service、可複用的邏輯
3. **前端現況**：現有的頁面、元件、可複用的 Composable
4. **依賴關係圖**：核心檔案之間的依賴關係
5. **實作模式**：從相似功能歸納出的標準實作流程
6. **缺口分析**：Story 需求與現有實作之間的差距，即「需要新做的部分」
