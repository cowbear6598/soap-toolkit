# Jira 子任務格式規範

建立子任務時，每個子任務需要 `summary` 和 `description` 兩個欄位。以下定義各類子任務的格式。

## 一般子任務

### Summary 格式

```
<功能描述> [$ARGUMENTS]
```

- `<功能描述>`：簡明扼要的任務名稱，以動作開頭（實作、建立、整合、串接等）
- `[$ARGUMENTS]`：原始 Story 的 Issue Key，用方括號包裹
- **嚴禁**加上任何非 Story Key 的標籤或前綴

範例：
```
完成註冊功能 [DCM-3999]
建立 User 資料表 [DCM-3999]
串接第三方支付服務 [DCM-3999]
```

### Description 格式

```
需求說明:
- <說明該子任務的具體目標和範圍>
- <列出需要實作的功能點>

參考檔案:
- <現有的相似實作，可作為開發參考的程式碼路徑>

依賴檔案:
- <該子任務會直接使用或呼叫的現有程式碼路徑>

新檔案:
- <預期需要新建的檔案路徑>
```

**欄位說明**：

| 欄位 | 用途 | 說明 |
|------|------|------|
| 需求說明 | 定義子任務範圍 | 讓開發者清楚知道要做什麼，列出具體的功能點 |
| 參考檔案 | 提供實作參照 | 現有 codebase 中結構相似的程式碼，開發者可以參照其模式來實作 |
| 依賴檔案 | 標示直接依賴 | 該子任務會 import、呼叫或繼承的現有檔案，開發時需要理解這些檔案 |
| 新檔案 | 預估新建檔案 | 根據分析，預期需要新建的檔案路徑，遵循現有專案的命名慣例 |

**重要規則**：
- 同一個檔案不能同時出現在「參考檔案」和「依賴檔案」中。如果一個檔案既可參考又被依賴，歸入「依賴檔案」（依賴層級高於參考）
- 所有檔案路徑指的是程式碼檔案，不包含 `.md` 等文件檔案
- 檔案路徑使用專案內的相對路徑
- 若某個欄位沒有內容，則省略該欄位（不要留空）

## HttpLog 子任務

### Summary
```
HttpLog
```

### Description
```
[]SIT
[]UAT
[]PROD
```

這是固定格式的 Checklist，用於追蹤各環境的 HttpLog 設定狀態。

## 文件更新子任務

### Summary
```
文件更新
```

### Description

留空。文件更新的具體內容在開發完成後才能確定。
