# 品質 Review 規則

## 檢查範圍

僅檢查本次有異動的程式碼。

## 檢查項目

### 單一職責

一個 class 或方法是否只做一件事。如果你很難用一句話描述它的職責，就代表它做太多了。

### 錯誤處理

- 不要用無意義的 try-catch 過度保護，try-catch 應該只出現在有實際失敗可能的邊界（第三方服務、I/O、網路請求）
- 其餘情況讓錯誤自然浮現，提前暴露問題比吞掉錯誤好
- 錯誤處理應該有明確的意圖，不是「以防萬一」

### 抽象合理性

- 不要為了一次性的操作建立抽象（不需要為只用一次的邏輯寫 helper）
- 反過來，如果同樣的邏輯出現多次，就該抽取
- 抽象的命名和層級應該讓使用者不需要看實作就能理解用途

### 資料流清晰度

- 資料的流向應該是可追蹤的，不要讓資料在多個地方被隱式修改
- 偏好不可變的資料結構，減少副作用
- 避免用全域狀態或隱式的依賴傳遞資料

### 非同步

- 偏好 async/await，避免阻塞式的等待
- 不要混用同步和非同步模式
