# 複雜度 Review 規則

## 檢查範圍

僅檢查本次有異動的程式碼。

## 檢查項目

### 主準則：SRP（Single Responsibility）

方法 / 類別應該能用一句話描述職責，描述中不應出現「以及」「並且」「然後」這類連接詞。
做不到 → 必須拆。**這是本面向最重要的判準。**

巢狀深度、參數數量、方法長度本身**不是**判準，它們只是 SRP 違反時常見的「症狀」。看到症狀時要先追因，確認真的違反 SRP 才回報，避免框架慣例被誤觸。

---

### 症狀：以下情況通常是 SRP 違反的徵兆，需要追因

#### 1. 巢狀過深（超過 3 層 if / loop / try）

先問：「是不是因為這個方法做了太多事？」

- ✅ 是 → 拆方法（從根本解 SRP），不要只用 early return 修表面
- ⚠️ 否（屬於框架慣例或結構性巢狀）→ **不算問題**，例如：
  - React component 的 conditional rendering
  - Go 的 `if err != nil` error handling 鏈
  - try-catch 包裹的多步驟流程
  - 已封裝的 state machine / parser

#### 2. 參數過多（超過 4 個基本型參數）

先問：「這些參數是否屬於同一個概念？」

- ✅ 是 → 封裝成 object（props / options / config）
- ⚠️ 否（已經是 options object、DTO、或語言慣例）→ **不算問題**

#### 3. 方法過長

**無硬性行數門檻。** 判準：能否用一句話描述職責。
做不到 → 拆。

#### 4. 混合抽象層級

同一方法混雜「高層流程控制」（驗證 → 處理 → 回傳）與「低層細節」（字串轉換、格式化、bit 操作）→ 應分離。

---

## 回報方式

發現問題時，說明：
- 為什麼這段程式碼違反 SRP（對應到哪個症狀）
- 具體在哪裡造成理解困難
- 提出多個重構方案，標註你最推薦的一個

## 輸出格式

```markdown
## 複雜度 Review 結果

若沒有任何問題，回報單行「無問題」即可，不列檔案清單或表格。

### ⚠️ 建議改善
| 檔案 | 行數 | 複雜類型 | 問題描述 | 建議方案 |
|------|------|----------|----------|----------|

### ❌ 必須重構
| 檔案 | 行數 | 複雜類型 | 問題描述 | 建議方案 |
|------|------|----------|----------|----------|
```

「複雜類型」欄位限填以下五種之一：

- `SRP 違反`（主準則，職責描述需要連接詞）
- `巢狀過深`（已排除框架慣例後仍判定為 SRP 問題）
- `參數過多`（已排除同概念封裝後仍判定為 SRP 問題）
- `方法過長`（無法一句話描述職責）
- `混合抽象層級`
